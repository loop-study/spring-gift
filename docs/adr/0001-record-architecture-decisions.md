# ADR-0001: ADR(Architecture Decision Record)을 채택한다

## Status
Accepted (2026-05-06)

## Context
spring-gift 과제는 구조 변경과 작동 변경을 분리하며 점진적으로 진행한다. 진행 과정에서 **트랜잭션 경계, 외부 API 호출 시점, 테스트 전략**처럼 코드만 봐서는 이유를 알 수 없는 결정들이 발생한다. 이 결정들은 다음과 같은 특성을 가진다.

- 선택지가 둘 이상이고 트레이드오프가 존재한다.
- 한 번 정하면 이후 변경이 그 결정을 따른다(예: "트랜잭션은 서비스 계층에서 시작한다").
- 결정의 근거가 PR 설명이나 커밋 메시지 안에 묻히면, 시간이 지난 뒤 재검토가 어렵다.

이 과제의 프로그래밍 요구 사항도 동일한 트리거를 명시하고 있어, 별도 형식으로 결정 기록을 남기기로 한다.

## Decision
저장소 내 `docs/adr/` 디렉터리에 **Michael Nygard 형식**의 ADR을 둔다.

- 파일명 규칙: `NNNN-kebab-case-title.md` (예: `0002-transaction-boundary-in-service-layer.md`)
- 4자리 zero-padded 일련번호. 한 번 발번된 번호는 **재사용하지 않는다**.
- Status는 `Proposed | Accepted | Deprecated | Superseded by ADR-XXXX` 중 하나.
- Status 변경 시 원본 ADR을 삭제하지 않는다 — 결정의 역사가 보존되어야 한다.

ADR을 남기는 트리거(다음 중 하나라도 해당하면 작성):
1. 선택지가 2개 이상이고 트레이드오프가 존재하는 경우
2. 반복적으로 따라야 할 규칙이나 경계를 정의한 경우
3. 테스트 전략이나 검증 방식이 결정의 핵심이었던 경우

ADR 본문은 다음 섹션을 갖는다.
`Status` / `Context` / `Decision` / `Consequences` / `Alternatives Considered`

## Consequences
- 결정의 **맥락(왜)** 이 코드 외부에 보존되어, 시간이 지나도 재검토 가능하다.
- ADR은 "결정의 스냅샷"이다. 사실이 바뀌어도 원본을 수정하지 않고 새 ADR로 supersede 한다 — 이를 통해 결정의 변천사가 추적된다.
- 사소한 변경까지 ADR을 남기면 노이즈가 된다. 위 트리거 조건에 해당하지 않으면 PR 설명/커밋 메시지로 충분하다고 본다.
- ADR 제목은 단정형으로 쓴다 ("X를 한다"). "X를 검토한다" 같은 모호한 표현은 피한다.

## Alternatives Considered
- **결정 사항을 PR 설명과 커밋 메시지에만 남긴다**
  - 거절. PR이 닫힌 후 검색이 어렵고, 여러 PR에 걸쳐 같은 규칙을 반복 설명하게 된다.
- **별도 위키나 외부 문서(Notion 등)에 둔다**
  - 거절. 코드와 동기화가 어렵다. ADR은 코드와 같은 저장소에 두어 PR 단위로 함께 리뷰되어야 한다.
- **결정을 코드 주석으로 남긴다**
  - 거절. 결정의 영향이 여러 파일에 걸쳐 있을 때 한 곳에서 표현하기 어렵고, 결정 자체와 그 결정을 따르는 코드의 구분이 흐려진다.
- **ADR 형식으로 MADR(Markdown ADR) 등 다른 템플릿 채택**
  - 거절. 이번 과제 규모에서는 Nygard 템플릿의 5개 섹션이 충분하고 가장 단순하다. 필요해지면 supersede ADR로 형식 자체를 바꿀 수 있다.
